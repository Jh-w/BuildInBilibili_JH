## 二、从市场视角发现的架构缺口

### 2.1 可观测性为零（🔴 最高优先级）

架构文档 §9（监控与可观测性）在 docs/ 版本有章节标题但内容不详，实际部署中完全缺失。

**市场为什么关心这个**：没有数据 = 无法复盘 = 无法做 Go/No-Go 决策。

| 缺口 | 为什么不只是技术问题 | 最小方案 |
|------|-------------------|---------|
| 无 PV/UV 统计 | 无法验证 SOM 预估（3-5 万 UV） | 前端 1 行 `fetch` 到计数端点 |
| 无分享卡片事件追踪 | 无法验证 K 因子（1.5-3.0） | Canvas `toBlob()` 后 `fetch` 到 /api/event |
| 无 API 调用量监控 | 无法评估成本和使用量 | DeepSeek 后台 dashboard（已有）+ 本地计数器 |
| 无错误率监控 | 无法评估降级策略是否生效 | FastAPI middleware 统计 5xx |

**建议**：在架构文档 §9 补充「最小可观测性方案」——3 个端点 + 1 个 local counter，总开发量 < 2 小时。

```
POST /api/event  { type: "page_view" | "share_card" | "chat_complete" }
GET  /api/stats   { pv, uv, shares, chats, errors }
```

### 2.2 容量规划缺失

架构文档 §7（部署方案）和 §8（性能优化）有单机性能指标（10 QPS），但缺少基于市场预期的容量校核。

| 场景 | 市场侧预估 | 架构侧是否扛得住 |
|------|-----------|----------------|
| 出分日峰值（6/23 下午 2-6 点） | 假设 50% UV 集中在 4 小时 → 峰值约 0.5-1 QPS | ✅ 10 QPS 容量足够 |
| 视频爆款流量冲击 | 单条视频 5 万播放 → 假设 5% 点击 → 2500 瞬时 UV | ✅ 静态前端走 CDN，API 异步 |
| DeepSeek API 并发限制 | 未知（DeepSeek 未公开并发上限） | ⚠️ 需测试——如果限制太低，高峰期可能排队 |

**建议**：架构文档补充「容量校核表」——基于 SOM 预估反推各层容量需求。

### 2.3 实际实现与 docs/ 版本不一致

docs/架构设计文档_v1.0.md（6/11）规划：
- 前端：React + TypeScript + Vite
- 部署：EdgeOne Pages + SCF

实际交付（6/23）：
- 前端：**Vanilla JS**（单文件 HTML，640 行）
- 部署：**裸机 uvicorn :8899**

**市场视角的影响**：
- ✅ Vanilla JS 更轻量 → 加载更快 → 转化率更高（移动端 4G 友好）
- ⚠️ 裸机部署 → 无 CDN → 如果视频爆款带来高并发，静态资源加载可能成为瓶颈（虽然 22KB 单文件影响极小）

**建议**：docs/ 版本标注「已偏离——详见根目录架构文档 v1.0」，或在 docs/ 版本中新增「实际交付 vs 原始规划」对比节。

### 2.4 System Prompt 的 A/B 测试能力

毒舌力度是产品的核心竞争力，但当前架构没有 A/B 测试能力：

| 场景 | 需求 | 当前能力 |
|------|------|---------|
| 高分段毒舌力度调优 | 对比 temperature 0.7 vs 0.85 的用户反馈 | ❌ 无法分流 |
| 低分段安慰语气调优 | 对比「鼓励开头」vs「直接给路径」的效果 | ❌ 无法对比 |
| 毒舌内容红线的漏网率 | 监测红线话题触发时 AI 是否仍越界 | ❌ 无监测 |

**建议**：在架构文档 V1.1 中补充「轻量 A/B 分流」方案——利用 `?variant=A` URL 参数选择不同 System Prompt 版本，前端随机分配。

### 2.5 国际化与多语言（远期）

架构文档未涉及。从市场视角，毒舌助手的中文「毒舌」文化是中国互联网特有的内容品类，短期内不需要国际化。但如果未来扩展为独立 IP，以下市场可能需要：

| 市场 | 适配难度 | 优先级 |
|------|---------|--------|
| 港澳台（繁体中文） | 低（翻译 + 本地学校数据） | P3 |
| 海外华人（SAT/ACT 场景） | 中（不同考试体系） | P4 |
| 日韩（类似应试文化） | 高（语言 + 考试体系完全不同） | P4 |

**建议**：远期考虑，但当前不投入。

## 三、架构与市场的接口问题

### 3.1 数据反馈循环断裂

```
理想循环：
  产品使用数据 → 市场策略调整 → 产品迭代 → 新数据
                                          
当前现状：
  产品使用数据 → ❌（不存在）→ 市场策略靠猜 → 产品无迭代依据
```

**架构侧最小修复**：上线 §2.1 中的 3 个端点，让市场侧有数据可读。

### 3.2 成本预警缺失

| 指标 | 预估 | 是否需要预警 |
|------|------|------------|
| DeepSeek API 日调用量 | 500-5000 次 | ✅ 如果单日超过 10000 次需要关注 |
| API 单日费用 | ¥5-50 | ✅ 如果单日 > ¥200 需排查（可能被刷） |
| 服务器 CPU/内存 | 极低（无状态） | 🟢 不是瓶颈 |

**建议**：架构文档补充「成本预警阈值」。
